# Shop Management POS

A desktop Point of Sale application for retail/communications shops, built with Electron + React.

---

## Prerequisites

- [Node.js](https://nodejs.org/) v18 or later
- npm (comes with Node.js)

---

## Running in Development

```bash
# Install dependencies (only needed once)
npm install

# Start both Vite dev server and Electron together
npm run dev
```

This runs two processes concurrently:
1. **Vite** dev server on `http://localhost:5173` (React hot reload)
2. **Electron** — waits for Vite to be ready, then loads the app from localhost

You can also run them separately:

```bash
# Only start the React dev server
npm run dev:react

# Only start Electron (requires Vite to already be running)
npm run dev:electron
```

---

## Building for Production

```bash
npm run build
```

This runs two steps in sequence:
1. `vite build` — compiles and bundles the React app into the `dist/` folder
2. `electron-builder` — packages Electron + the `dist/` output into a Windows installer

The installer is output to:
```
dist-electron/
  Shop Management POS Setup x.x.x.exe   ← NSIS installer
```

Install this `.exe` on any Windows machine to deploy the app.

### Lint before building

```bash
npm run lint
```

ESLint with zero warnings allowed. Fix all warnings before building for production.

## Where Data is Stored

All data is stored in a single JSON file on the machine where the app is installed:

```
C:\ProgramData\pos_data\config.json
```

This file is created automatically on first launch. You can open it in any text editor to inspect or manually edit data.

### Keys stored in config.json

| Key              | Type    | Description                                               |
|------------------|---------|-----------------------------------------------------------|
| `inventory`      | Array   | All products with stock levels, variants, and cost prices |
| `orders`         | Array   | Completed and cancelled sales (full order history)        |
| `stockLogs`      | Array   | Every stock addition (purchase log)                       |
| `credits`        | Array   | Credit sale records with payment history                  |
| `settings`       | Object  | Store name, address, phone, GSTIN, logo (base64), etc.    |
| `users`          | Array   | Staff accounts created by the administrator               |
| `adminPassword`  | String  | Administrator login password (only written after change)  |
| `invoiceCounter` | Number  | Auto-incrementing invoice sequence number (never resets)  |

> **Backup:** Copy `C:\ProgramData\pos_data\config.json` to back up all data.
> **Transfer to new machine:** Copy the same file to `C:\ProgramData\pos_data\config.json` on the new machine before launching the app.

## Architecture

### How Electron and React Connect

```
┌─────────────────────────────────┐       ┌──────────────────────────────┐
│        Electron Main Process     │       │    React Renderer Process    │
│         (electron/main.js)       │       │       (src/ via Vite)        │
│                                  │  IPC  │                              │
│  electron-store                  │◄─────►│  ShopContext / AuthContext   │
│  C:\ProgramData\pos_data\        │       │  window.require('electron')  │
│    config.json                   │       │  ipcRenderer.invoke/send     │
└─────────────────────────────────┘       └──────────────────────────────┘
```

The React app communicates with the main process via Electron IPC:
- `ipcRenderer.invoke('get-data', key)` — reads a key (returns a Promise)
- `ipcRenderer.send('save-data', key, value)` — writes a key (fire-and-forget)

This is possible because `nodeIntegration: true` and `contextIsolation: false` are set in the BrowserWindow config — an intentional simplification for a local-only desktop app.

### Authentication Flow

```
App starts
    │
    ▼
sessionStorage has 'auth_user'?
    ├── No  → Show Login screen
    └── Yes → Load AppContent
                  │
                  ▼
            isAdmin?
            ├── Yes → AdminDashboard + Settings + UserManagement
            └── No  → Full POS interface (Dashboard, POS, Orders, Products, Credits, Settings)
```

There are two roles:

**Administrator**
- Username: `Administrator` (hardcoded, case-sensitive)
- Default password: `Password123` (stored in `adminPassword` key after first change)
- Can: manage staff accounts, change admin password, view revenue overview
- Cannot: access POS, orders, inventory, or credits

**Staff**
- Created by the Administrator via the Users page
- Can have an optional expiry date or deactivation date
- Access to all POS operations

Session is stored in `sessionStorage` (cleared when the browser/app window closes).

### State Management

All business data lives in `ShopContext` (React Context + electron-store sync):

```
ShopContext
├── inventory    — product catalog (with variants, stock, avg cost)
├── orders       — all sales
├── cart         — current POS session (in-memory only, not persisted)
├── stockLogs    — purchase/stock-in history
├── credits      — credit sale records
├── settings     — store configuration
└── isLoading    — true until all data is loaded from electron-store on startup
```

Key mutation functions:
- `saveProductWithVariants` — create/update a product with multiple variants in one atomic call
- `editProductFull` — edit product metadata and add stock to variants
- `processSale(cartItems, discountDetails, paymentInfo, invoiceId)` — deducts stock, creates order record, clears cart
- `cancelOrder(orderId)` — marks order cancelled and restocks all non-variable-price items
- `addCredit(creditData)` — creates a credit sale record linked to an order
- `recordCreditPayment(creditId, amount, note)` — records a partial/full payment; auto-settles when fully paid

### Product & Variant Model

Every product has one or more variants stored as a map:

```json
{
  "id": 1234567890,
  "name": "Airtel SIM",
  "isRecharge": true,
  "isVariablePrice": false,
  "stock": 50,
  "avgCostPrice": 18.5,
  "price": 25,
  "gstRate": 18,
  "minStockAlert": 5,
  "variants": {
    "Prepaid": { "quantity": 30, "avgCost": 18, "price": 25, "gstRate": 18, "code": "ATL-PRE", "color": "#FF0000", "description": "" },
    "Postpaid": { "quantity": 20, "avgCost": 19, "price": 30, "gstRate": 18, "code": "ATL-POST", "color": "#0000FF", "description": "" }
  }
}
```

When stock is added, the per-variant weighted average cost is recalculated:
```
new avgCost = (oldQty × oldAvgCost + addedQty × newUnitCost) / (oldQty + addedQty)
```

The master product `avgCostPrice` and `stock` are always re-derived from the sum of all variants.

---

### Invoice Numbering

Invoice IDs are formatted as `DDMMYY-000001` and use a persistent counter stored under `invoiceCounter` in config.json. The counter never resets — it only increments.

### Order Model

```json
{
  "id": "180326-000001",
  "date": "2026-03-18T10:30:00.000Z",
  "status": "completed",
  "paymentMethod": "cash",
  "subtotal": 500,
  "totalGst": 90,
  "discount": { "type": "percent", "value": 10, "amount": 59 },
  "finalTotal": 531,
  "tenderedAmount": 600,
  "changeGiven": 69,
  "items": [
    {
      "id": 1234,
      "name": "Airtel SIM",
      "variantName": "Prepaid",
      "quantity": 2,
      "price": 25,
      "gstRate": 18,
      "itemTotal": 50,
      "itemGst": 9,
      "costAtSale": 18
    }
  ]
}
```

`costAtSale` captures the variant's `avgCost` at the moment of sale — used for profit calculations on the dashboard.

### Credit Sale Flow

```
POS checkout → customer pays less than finalTotal
    │
    ▼
processSale() — creates the order as normal (status: 'completed')
    │
    ▼
addCredit() — creates a credit record:
    { totalAmount, paidAmount, pendingAmount, status: 'pending', payments: [] }
    │
    ▼
Credits page → recordCreditPayment(creditId, amount)
    │
    ▼
pendingAmount reaches 0 → status changes to 'settled'
```

### Dashboard Periods

The staff dashboard supports five time periods selectable via a tab toggle:

| Period  | Covers                              |
|---------|-------------------------------------|
| Today   | From midnight of current day        |
| Week    | Last 7 days                         |
| Month   | From the 1st of the current month   |
| Quarter | From the 1st of the current quarter |
| Year    | From January 1st of current year    |

Stats shown: Revenue, Orders, Cash/UPI/Card collected, Money Invested, Profit, Margin, Total Products, Low Stock, Credit Due, Credit Collected.

Profit and Margin only appear when products have `avgCostPrice` set (i.e., stock was added with a buying cost).
